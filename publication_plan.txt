Plan działania - publikacja na pub.dev

  Faza 1: Przygotowanie podstawowe

  1. Wybór licencji - dodać właściwą licencję (MIT/BSD/Apache 2.0)
  2. Decyzja: Flutter vs Pure Dart package
    - Obecnie: Flutter package
    - Rekomendacja: Pure Dart (szersze zastosowanie - CLI, backend, web)
    - Jeśli Flutter nie jest potrzebny, usunąć zależność
  3. Aktualizacja pubspec.yaml
    - Opis paczki (description)
    - Homepage/repository URL
    - Topics dla lepszego discovery
    - Poprawna wersja początkowa (0.1.0 lub 1.0.0)

  Faza 2: Dokumentacja

  4. Dartdoc comments dla:
    - Klasy Parsable
    - Metody get<T>
    - Statycznych metod konfiguracyjnych
    - Typedef'ów
  5. README.md z:
    - Zwięzłym opisem problemu i rozwiązania
    - Sekcją Features
    - Installation instructions
    - Przykładami użycia (Twoje Owner/OwnerConfig to świetny przykład!)
    - API reference
    - Badges (pub.dev version, license)
  6. CHANGELOG.md - opis pierwszej wersji

  Faza 3: Testy

  7. Unit testy pokrywające:
    - Podstawowe parsowanie typów prostych (int, String, bool, double)
    - Parsowanie null values
    - Numeric conversions
    - Zagnieżdżone obiekty z parserem
    - Error handling (brak parsera, błędny typ)
    - Equatable functionality
    - Edge cases (puste mapy, null data)

  Faza 4: Przykłady

  8. Katalog /example z:
    - example/parsable_example.dart - podstawowe użycie
    - Kompletny przykład z Owner/OwnerConfig
    - README w example/ wyjaśniające przykłady

  Faza 5: Jakość kodu

  9. Analysis options - upewnić się że flutter analyze przechodzi
  10. Formatting - dart format .
  11. Pub.dev score optimization:
    - 100% documentation coverage
    - Passing static analysis
    - Multi-platform support

  Faza 6: Publikacja

  12. Dry run: dart pub publish --dry-run
  13. Weryfikacja score: sprawdzić wszystkie wymagania pub.dev
  14. Publikacja: dart pub publish
  15. Post-publish:
    - Weryfikacja strony na pub.dev
    - Sprawdzenie przykładów w dokumentacji
    - Monitoring score i sugestii pub.dev

  Faza 7: Opcjonalne ulepszenia (po publikacji)

  16. CI/CD - GitHub Actions dla testów
  17. Coverage badge - codecov/coveralls
  18. Null safety verification - upewnić się że jest w 100% null-safe
  19. Performance benchmarks - jeśli istotne
  20. Dodatkowe features (do rozważenia):
    - Lista/array parsing z parserem elementów
    - Nested path support (get('user.address.city'))
    - Validation hooks
    - JSON serialization support

  ---
  Kluczowe decyzje do podjęcia:

  1. Flutter vs Pure Dart - co preferujesz?
  2. Licencja - MIT (najpopularniejsza), BSD, Apache 2.0?
  3. Repository - GitHub URL gdzie będzie kod?
  4. Wersja początkowa - 0.1.0 (beta) czy 1.0.0 (stable)?
  5. Nazwa - "parsable" jest dostępna na pub.dev?
